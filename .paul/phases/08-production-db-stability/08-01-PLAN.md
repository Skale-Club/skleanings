<objective>
## Goal
Fix Vercel 504 timeouts by correcting the DB connection priority in serverless mode and hardening the most frequently fired admin query.

## Purpose
Production is unusable — every DB-touching authenticated endpoint returns 504 after 30 seconds. Root cause identified: `server/db.ts` uses `POSTGRES_URL_NON_POOLING` (direct TCP connection, no pooler) in serverless mode instead of `POSTGRES_URL` (pgBouncer pooler). When Neon wakes from auto-pause or connections compete, direct TCP connections hang until Vercel's 30s function timeout fires. Switching to the pooler URL eliminates the hang.

## Output
- `server/db.ts` — corrected connection priority + shorter connection timeout
- `CalendarReconnectBanner.tsx` — query no longer refetches on every window focus (which was causing repeated 504 storms on tab switch)
</objective>


```gherkin

```

```gherkin

```

    ```typescript
      : process.env.DATABASE_URL || process.env.POSTGRES_URL || process.env.POSTGRES_URL_NON_POOLING || "";
    ```

    Also reduce `connectionTimeoutMillis` from 30000 to 8000 in the Pool config (line 39):
    ```typescript
    connectionTimeoutMillis: 8000,
    ```

    REASON: `POSTGRES_URL` is Vercel Postgres's pgBouncer pooler — persistent connections, no TCP handshake per request. `POSTGRES_URL_NON_POOLING` creates a direct TCP connection from the serverless function to Neon; when Neon auto-pauses, this hangs for up to 30s. Reducing timeout to 8s ensures the function returns an error before Vercel's 30s hard kill.

    Do NOT change: the pool `max`, `idleTimeoutMillis`, SSL config, or error handler.
