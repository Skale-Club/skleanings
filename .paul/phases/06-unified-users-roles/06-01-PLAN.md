<objective>
## Goal
Add `role` column to `users` table, `phone` column, `userId` FK on `staffMembers`, update auth middleware to check role from DB (not env var), and expose role in AuthContext so frontend can route by role.

## Purpose
Foundation for the entire v0.6 milestone. Currently auth checks `ADMIN_EMAIL` env var — this plan replaces that with a DB-driven `role` column. All subsequent plans (unified users page, staff settings) depend on this.

## Output
- `shared/schema.ts` — `users` table gets `role` + `phone` columns; `staffMembers` gets `userId` FK
- `server/lib/auth.ts` — `requireAdmin`, `requireUser`, `requireAuth` middlewares checking DB role
- `server/storage.ts` — `getUserBySupabaseId` method for auth lookup
- `client/src/context/AuthContext.tsx` — exposes `role` (fetched from DB via API)
- Supabase migration for new columns
</objective>

<task type="auto">
  <name>Task 2: Auth middleware refactor — role-based</name>
  <files>server/lib/auth.ts, server/storage.ts</files>
  <action>
    **Refactor `server/lib/auth.ts`:**

    1. Import `storage` from `../storage`.

    2. Create helper `getAuthenticatedUser(req)`:
       - Extract Bearer token from Authorization header
       - Call `supabase.auth.getUser(token)` to validate
       - If valid, look up our DB user: `storage.getUserByEmail(supabaseUser.email)`
       - If DB user not found, return null (user exists in Supabase auth but not in our users table)
       - Attach both to req: `(req as any).user = dbUser; (req as any).supabaseUser = supabaseUser;`
       - Return the DB user (which now has `role`)

    3. **`requireAuth`** — any authenticated user:
       - Call `getAuthenticatedUser(req)`
       - If null → 401
       - Proceed (admin, user, or staff all pass)

    4. **`requireUser`** — admin or user role:
       - Call `getAuthenticatedUser(req)`
       - If null → 401
       - If role !== 'admin' && role !== 'user' → 403 "Insufficient permissions"
       - Proceed

    5. **`requireAdmin`** — admin role only:
       - Call `getAuthenticatedUser(req)`
       - If null → 401
       - If role !== 'admin' → 403 "Admin access required"
       - Proceed
       - Remove the `ADMIN_EMAIL` env var check — role is now the source of truth

    6. Export all three: `requireAuth`, `requireUser`, `requireAdmin`.

    **Add API endpoint `GET /api/auth/me`:**
    In `server/routes/staff.ts` is wrong place — add a new route or use existing user routes.
    Check: does a user routes file exist?

    Actually, add it directly in `server/lib/auth.ts` as an exported handler, or better: create a small endpoint in the existing routes setup. For simplicity, add to the Express app setup in index.ts or wherever routes are mounted. 

    The simplest approach: export a `getAuthMe` handler from auth.ts that the route setup can mount:
    ```typescript
    export async function getAuthMe(req: Request, res: Response) {
      const authHeader = req.headers.authorization;
      if (!authHeader?.startsWith('Bearer ')) return res.status(401).json({ message: 'Not authenticated' });
      const token = authHeader.split('Bearer ')[1];
      const { data: { user: supabaseUser }, error } = await supabase.auth.getUser(token);
      if (error || !supabaseUser) return res.status(401).json({ message: 'Not authenticated' });
      const dbUser = await storage.getUserByEmail(supabaseUser.email!);
      if (!dbUser) return res.status(404).json({ message: 'User not found' });
      res.json({ id: dbUser.id, email: dbUser.email, role: dbUser.role, firstName: dbUser.firstName, lastName: dbUser.lastName, profileImageUrl: dbUser.profileImageUrl, phone: dbUser.phone });
    }
    ```

    Mount as `GET /api/auth/me` in the routes setup file.

    Avoid: Breaking existing `requireAdmin` usage on all routes — the new version must export the same function name with the new behavior.
  </action>
