# Milestone v15.0: Tenant Onboarding Experience

**Status:** ✅ SHIPPED 2026-05-14
**Phases:** 55–56
**Total Plans:** 5

## Overview

New tenants receive verification and welcome emails on signup, unverified admins see a persistent banner with a resend link, and a live setup checklist guides them through the minimum configuration needed to go live.

## Phases

### Phase 55: Email Verification + Welcome Email

**Goal**: New tenants receive a verification email immediately after signup and a welcome email with first-steps guidance — unverified admins see a persistent banner in the admin panel until they verify their email address
**Depends on**: Phase 54
**Plans**: 3 plans

Plans:

- [x] 55-01: email_verification_tokens Supabase migration + users.emailVerifiedAt + Drizzle schema + 4 IStorage methods (createEmailVerificationToken, findEmailVerificationToken, markEmailVerificationTokenUsed, setEmailVerified)
- [x] 55-02: buildVerificationEmail + buildWelcomeEmail email builders + fire-and-forget sends in signup.ts + GET /api/auth/verify-email + POST /api/auth/resend-verification routes
- [x] 55-03: VerifyEmail.tsx public page + /verify-email route in App.tsx + admin-me emailVerifiedAt extension + AdminTenantAuthContext emailVerifiedAt + dismissible yellow banner in Admin.tsx

### Phase 56: Setup Checklist

**Goal**: New tenant admins see a live setup checklist in their dashboard that tracks the three minimum actions needed to go live — the checklist is driven by real DB state and can be dismissed once complete or manually by the admin
**Depends on**: Phase 55
**Plans**: 2 plans

Plans:

- [x] 56-01: setup_dismissed_at Supabase migration + Drizzle schema + server/routes/admin-setup.ts (GET /api/admin/setup-status + POST /api/admin/setup-dismiss) mounted in routes.ts
- [x] 56-02: useSetupStatus React Query hook + SetupChecklist.tsx component (3 items, dismiss) + Admin.tsx wiring above dashboard section

## Key Decisions

- Email verification tokens use global registry pattern (db directly, no tenantId scope) — same as passwordResetTokens from Phase 47
- GET /api/auth/verify-email is a server-side redirect route (success → /admin, failure → /verify-email?error=invalid) — frontend VerifyEmail.tsx only handles the error case
- admin-me made async to include emailVerifiedAt from DB lookup — session alone doesn't have this field
- SetupChecklist hides when dismissed=true OR all 3 items complete — driven by React Query with 10s staleTime
- adminSetupRouter uses res.locals.storage exclusively — no global singleton import
