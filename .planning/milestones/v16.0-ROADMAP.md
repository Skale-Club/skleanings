# Milestone v16.0: Staff Invitation Flow

**Status:** ✅ SHIPPED 2026-05-15
**Phases:** 57–58
**Total Plans:** 5

## Overview

Tenant admins can invite staff members by email — no super-admin intervention required. Staff accept the invite via a token-secured link, set their password, and are immediately logged in and associated with the tenant.

## Phases

### Phase 57: Staff Invitation Backend

**Goal**: Tenant admins can invite staff members by email via a token-secured flow — the invitation is persisted, a branded email is sent, and all token lifecycle operations (validate, accept, revoke) are exposed as API endpoints
**Depends on**: Phase 56
**Plans**: 3 plans

Plans:

- [x] 57-01: staff_invitations Supabase migration + Drizzle schema + 5 IStorage methods (createStaffInvitation, findStaffInvitation, markInvitationAccepted, revokeStaffInvitation, getPendingInvitations)
- [x] 57-02: buildInviteEmail() + POST /api/admin/staff/invite (fire-and-forget Resend) + DELETE /api/admin/staff/invite/:id (409 if accepted) + GET /api/admin/staff/invitations
- [x] 57-03: GET /api/auth/validate-invite (public, 410 for invalid) + POST /api/auth/accept-invite (atomic db.transaction creating users + user_tenants, sets session, returns adminUrl)

### Phase 58: Staff Invitation Frontend

**Goal**: Staff complete account setup via a public accept-invite page; admins send and manage pending invitations from /admin/users
**Depends on**: Phase 57
**Plans**: 2 plans

Plans:

- [x] 58-01: AcceptInvite.tsx public page (3-state machine: validating/expired/form) + /accept-invite route in App.tsx + cross-subdomain redirect on 201
- [x] 58-02: PendingInvitationsSection component (React Query + Invite dialog + AlertDialog confirm Revoke) mounted in UnifiedUsersSection

## Milestone Summary

**Files changed:** ~12 files across 2 phases

**Key decisions:**
- Staff invitations use global registry IStorage methods (db directly, no this.tenantId) — same pattern as passwordResetTokens, emailVerificationTokens
- POST /accept-invite uses atomic db.transaction for users + user_tenants creation, mirrors signupTenant pattern
- DELETE invite uses direct db query for 409 (acceptedAt IS NOT NULL) detection since `findStaffInvitation` filters out accepted invitations
- Frontend uses `fetch` with `credentials: 'include'` (NOT apiRequest from queryClient) to match Phase 57's requireAdmin session-cookie auth
- PendingInvitationsSection mounted in UnifiedUsersSection alongside UsersSection — matches existing admin IA at /admin/users
