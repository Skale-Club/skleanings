**Change B — Gate the DB-miss branch BEFORE setting the cache:**

After the `if (!row) { res.status(404)...; return; }` block, and BEFORE the `hostnameCache.set(hostname, tenant)` call (line 46), insert the verification gate:

```typescript
// Phase 61 CD-06: non-primary domains must be DNS-verified to resolve.
// Primary *.xkedule.com subdomains bypass this check (the 61-01 backfill set
// verified=true for all existing primaries, and new signups create primaries
// via storage.signupTenant — which still uses addDomain(..., isPrimary=true)
// without a verification token; those rows would be verified=false, so we
// special-case isPrimary here rather than tightening signupTenant).
if (!row.isPrimary && !row.verified) {
  res.status(404).json({ message: "Unknown tenant" });
  return;  // do NOT cache — unverified entries are not servable
}

tenant = row.tenant;
hostnameCache.set(hostname, tenant);
```

Replace the existing `tenant = row.tenant; hostnameCache.set(hostname, tenant);` lines accordingly. **The key invariant:** anything in `hostnameCache` is safe to serve. We never cache unverified rows, so the cache-hit path needs no additional check.

**Change C — Keep the existing 503/402 logic AS-IS.** Do not touch:
- `if (tenant.status === 'inactive')` — line 49
- Subscription/402 enforcement — lines 54-76
- The final `res.locals.tenant = tenant; res.locals.storage = DatabaseStorage.forTenant(tenant.id); next();`
- The outer try/catch + `next(err)`

**Why this design over the alternative ("cache and check"):**
- Caching `{ tenant, verified, isPrimary }` and re-checking on every hit would expand the cache type for one extra boolean comparison per request — cheap, but only useful if verified state can flip while cached. With this design, `invalidateTenantCache` (already called by 61-02 on verify success) ensures the next request reloads from DB and now passes the gate. Same end behavior, simpler invariant: "if it's in the cache, it's servable."
- Note: when a domain is *unverified* it never enters the cache, so verification-success is a guaranteed cold lookup (correct). When a domain is *deleted*, 61-02 also calls `invalidateTenantCache` — next request misses cache, misses DB, returns 404 (correct).

**Edge cases verified by this design:**
1. Existing primary `{slug}.xkedule.com` → 61-01 backfill set verified=true → either branch (primary OR verified) passes → cached → resolves. ✓
2. New signup primary (via `storage.signupTenant`, server/storage.ts:2598) → isPrimary=true, verified=false → `!row.isPrimary` is false → gate passes → cached. ✓ (the isPrimary clause is doing the work here, intentional)
3. Newly-added custom domain (via 61-02 POST) → isPrimary=false, verified=false → gate triggers 404. ✓
4. Just-verified custom domain → 61-02 called invalidateTenantCache → next request cold-loads → verified=true → cached → resolves. ✓
5. Deleted custom domain → invalidateTenantCache called → next request misses cache, DB row gone → 404 via existing `!row` branch. ✓

**Do NOT:**
- Tighten the message to differentiate "unverified" from "unknown" — leaking that distinction would let attackers enumerate registered-but-unverified domains. The phase context explicitly says "treat as unknown hostname (404)."
- Add a separate cache for unverified domains — out of scope, no value.
- Change the `req.hostname` source (per Phase 40 decision in STATE.md: req.hostname avoids port contamination).
- Touch the imports — `domains` and `tenants` are already imported (line 3).
