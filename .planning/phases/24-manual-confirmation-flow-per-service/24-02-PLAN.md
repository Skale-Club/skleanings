<objective>
Implement the server-side logic for the manual confirmation flow:
1. `POST /api/bookings` — detect if primary service requires confirmation and set `awaiting_approval` status accordingly
2. `PUT /api/bookings/:id/approve` — admin endpoint to approve (set confirmed)
3. `PUT /api/bookings/:id/reject` — admin endpoint to reject (set cancelled, optional reason stored as notes via update)
4. Notification: fire `booking_awaiting_approval` trigger via both Twilio and Telegram when booking lands in awaiting_approval

Purpose: Without this, the schema change in Plan 01 has no behavioral effect. This plan wires requiresConfirmation into the booking creation flow and gives admin the tools to act on it.

Output: Updated `server/routes/bookings.ts`, `server/lib/notification-templates.ts`, `server/integrations/twilio.ts`, `server/integrations/telegram.ts`.
</objective>

      try {
        const booking = await storage.updateBookingStatus(Number(req.params.id), 'confirmed');

    // Reject a booking awaiting confirmation — sets status to cancelled
    // Optional: { reason: string } in body, stored via generic updateBooking
    router.put('/:id(\\d+)/reject', requireAdmin, async (req, res) => {
      try {
        const id = Number(req.params.id);
        await storage.updateBookingStatus(id, 'cancelled');
        // If a reason was provided, update notes via a generic patch
