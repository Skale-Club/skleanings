The token markers (must appear in client/index.html exactly as spelled). Note the og:image and twitter:image are now WHOLE-TAG block tokens per D-07 / RESEARCH Open Question 1:
- Plain attribute tokens: `{{SEO_TITLE}}`, `{{SEO_DESCRIPTION}}`, `{{CANONICAL_URL}}`, `{{OG_TYPE}}`, `{{OG_SITE_NAME}}`, `{{OG_LOCALE}}`, `{{TWITTER_CARD}}`, `{{TWITTER_SITE}}`, `{{TWITTER_CREATOR}}`, `{{ROBOTS}}`, `{{COMPANY_NAME_ALT}}`, `{{JSON_LD}}`
- Whole-element block tokens (emitted as full `<meta>` tag string OR empty string): `{{OG_IMAGE_BLOCK}}` (covers og:image + og:image:alt), `{{TWITTER_IMAGE_BLOCK}}` (covers twitter:image + twitter:image:alt)

Vite Pitfall (RESEARCH.md § Common Pitfalls 1-2):
- Place tokens ONLY in attributes Vite does not transform: content="" of meta, href="" of link rel="canonical"/"alternate", text content of <title> and <script type="application/ld+json">.
- DO NOT tokenize src="" or rel="icon" href="" — leave favicon/loader.css/main.tsx as static paths.
- ALWAYS run injectSeoMeta AFTER vite.transformIndexHtml (so HMR scripts already injected).
- Block tokens like `{{OG_IMAGE_BLOCK}}` placed on their own line between other meta tags are safe — they expand to either a full `<meta>` element or empty string.

Vercel filesystem precedence (RESEARCH.md § Common Pitfalls 6):
- Vercel evaluates filesystem BEFORE rewrites — `/assets/*`, `/favicon.png`, `/loader.css` are CDN-served regardless of the catch-all `/(.*)` rewrite. NO need to add explicit `/assets/(.*)` rewrites.
- The catch-all only matches genuinely-missing-file paths (`/`, `/services`, `/booking/...`).
