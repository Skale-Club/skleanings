<objective>
Deliver the tenant-facing `/admin/payments` page that lets a tenant admin connect a Stripe Express account, monitor capability flags, and refresh status after returning from Stripe-hosted onboarding (SC-06).

Purpose: Without this UI, a tenant admin has no way to initiate Stripe Connect onboarding — they would need direct API/curl access to call `/api/admin/stripe/connect/onboard`. This plan turns the Phase 63 backend into a usable feature.

Output: New `PaymentsSection` component, AdminSection union extended with `'payments'`, sidebar entry with Wallet icon, and inline render in Admin.tsx when `activeSection === 'payments'`.
</objective>

    Create the file `client/src/components/admin/PaymentsSection.tsx`. Export a default React component `PaymentsSection` (no props — it self-fetches via React Query, mirroring `BillingPage` pattern but as an inline section instead of a standalone page).

    Required behavior:

    1. **Status query** — `useQuery<StripeConnectStatus>` with queryKey `['/api/admin/stripe/status']`. Fetch via `fetch('/api/admin/stripe/status', { credentials: 'include' })`. Define the response type locally:
       ```ts
       interface StripeConnectStatus {
         connected: boolean;
         stripeAccountId: string | null;
         chargesEnabled: boolean;
         payoutsEnabled: boolean;
         detailsSubmitted: boolean;
       }
       ```

    2. **Render** a single Card titled "Stripe Payments" / description "Connect your Stripe account to receive customer payments directly." Inside CardContent show:
       - **Connection state badge**:
         - `status.connected === true` → green Badge "Connected" (`bg-green-100 text-green-800`)
         - Else → gray Badge "Not Connected" (`bg-gray-100 text-gray-600`)
       - **Stripe Account ID row** (only when connected): label "Account ID" + value in mono font (`font-mono text-sm`)
       - **Three capability rows**, each with a label and a check (`Check` lucide icon, green) or x (`X` lucide icon, gray):
         - "Charges Enabled" — bound to `status.chargesEnabled`
         - "Payouts Enabled" — bound to `status.payoutsEnabled`
         - "Details Submitted" — bound to `status.detailsSubmitted`
       - Render with the same row layout used in BillingPage features section (label on left, glyph on right).

    3. **Primary button** — single button whose label depends on state:
       - `!status.connected` → "Connect Stripe Account"
       - `status.connected && !status.detailsSubmitted` → "Continue Onboarding"
       - `status.connected && status.detailsSubmitted` → "Update Stripe Account"

       Style: brand yellow CTA (`className="rounded-full bg-[#FFFF01] text-black font-bold hover:bg-yellow-300"`).

       On click: call `useMutation` whose `mutationFn` POSTs to `/api/admin/stripe/connect/onboard` (credentials: 'include', method: 'POST'). On success the response is `{ url }` — set `window.location.href = data.url`. On error show a toast with the error message. Disable the button while `mutation.isPending` and show "Redirecting to Stripe…".

    4. **Refresh Status button** — secondary outline button labeled "Refresh Status" with `RefreshCw` icon. On click: POST to `/api/admin/stripe/refresh`; on success call `queryClient.invalidateQueries({ queryKey: ['/api/admin/stripe/status'] })`. Handle 404 specifically (when no account exists yet) — show toast "No Stripe account yet — connect first." rather than treating as a hard error. Hide this button entirely when `!status.connected` (refresh is meaningless without an account).

    5. **Return-from-Stripe handling** — on component mount, read `?status=success` from `window.location.search` via `new URLSearchParams(window.location.search)`. If present:
       - Show toast `{ title: 'Returned from Stripe', description: 'Refreshing status…' }`
       - Immediately trigger the refresh mutation
       - Strip the query param via `window.history.replaceState({}, '', '/admin/payments')` to prevent the toast from re-firing on remount
       Use `useEffect` with empty dependency array — run once.

    6. **Loading + error states** — while `query.isLoading`, show a Card with a skeleton or "Loading payment status…" text. On `query.isError`, show a red error message inside the Card with the error text. Do not block other UI.

    7. **No props, no getAccessToken** — this section is rendered inside Admin.tsx tenant-auth gated wrapper; session cookie handles auth. Use `credentials: 'include'` on all fetches.

    Reference styling and structure from `client/src/pages/admin/BillingPage.tsx` (Card layout, Badge classes, brand button class) but render as a section (no `max-w-2xl mx-auto py-12 px-4` outer wrapper — Admin.tsx provides the padding wrapper).
